# V2Plugin.Decrypt panics on empty ciphertext (Remote DoS)

### VULNERABILITY INFORMATION

**Component:** pkg/plugin/plugin_v2.go:182
**Affected Version:** aws-encryption-provider @ 4341c70 (all versions)
**Found by:** Fuzz testing (FuzzV2Decrypt, seed#0: empty []byte{})
**TLP:** TLP:Amber

---

## Summary

`V2Plugin.Decrypt()` accesses `request.Ciphertext[0]` without checking that the slice is non-empty. Same root cause as BUG-001 but in the V2 (KMS v2) API path. Crashes the entire gRPC server process.

---

## Vulnerable Code

```go
// pkg/plugin/plugin_v2.go:178-189
func (p *V2Plugin) Decrypt(ctx context.Context, request *pb.DecryptRequest) (*pb.DecryptResponse, error) {
    zap.L().Debug("starting decrypt operation")

    startTime := time.Now()
    storageVersion := kmsplugin.KMSStorageVersion(request.Ciphertext[0])  // <-- LINE 182: PANIC
    switch storageVersion {
    case kmsplugin.KMSStorageVersionV2:
        request.Ciphertext = request.Ciphertext[1:]
    default:
        return nil, fmt.Errorf("version %s in Ciphertext doesn't match kmsplugin", storageVersion)
    }
```

## Root Cause

Identical to BUG-001. The `KMSStorageVersion` type conversion indexes `request.Ciphertext[0]` without a length check. The gRPC `DecryptRequest.Ciphertext` field is a `[]byte` that can be empty or nil.


Minimal gRPC client reproducer:
```go
conn, _ := grpc.Dial("unix:///var/run/kmsplugin/socket.sock", grpc.WithInsecure())
client := v2pb.NewKeyManagementServiceClient(conn)
// This crashes the server:
client.Decrypt(context.Background(), &v2pb.DecryptRequest{Ciphertext: []byte{}})
```


## Suggested Fix

```go
func (p *V2Plugin) Decrypt(ctx context.Context, request *pb.DecryptRequest) (*pb.DecryptResponse, error) {
    zap.L().Debug("starting decrypt operation")

    startTime := time.Now()
    if len(request.Ciphertext) == 0 {
        return nil, fmt.Errorf("invalid empty ciphertext")
    }
    storageVersion := kmsplugin.KMSStorageVersion(request.Ciphertext[0])
```

## Platform

Tested on macOS arm64, Go 1.26.1. Bug is platform-independent (pure Go).

## Impact

## Impact

Same as another my bug https://hackerone.com/reports/3620748: unrecoverable server crash, Kubernetes Secrets unavailable during restart. Both V1 and V2 APIs are vulnerable — a single server process hosts both, so crashing via either API takes down both.

- **Server crash:** The panic is unrecovered — it kills the entire aws-encryption-provider process
- **Kubernetes secret unavailability:** While the pod restarts, kube-apiserver cannot encrypt/decrypt any Secrets at rest, blocking pod creation, configmap reads, service account token issuance
- **No authentication required:** The gRPC server has no auth — any process with access to the Unix socket can trigger this
- **Trivially exploitable:** Single gRPC call with empty ciphertext, no valid KMS credentials needed
- **Affects both nil and empty slices:** Both `Cipher: nil` and `Cipher: []byte{}` trigger the panic
